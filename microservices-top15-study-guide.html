01

Service Decomposition

Design

Instead of building one giant application, you break the system into small services, one per business capability. Each service does one job, has its own team, its own code, and can be deployed on its own.

Real-life analogy
🍽️ A restaurant kitchen. A small food stall has one cook doing everything — starters, main course, dessert, billing. If he falls sick, the stall shuts. A big restaurant has separate stations: grill, dessert, drinks, billing. Each station works independently. If the dessert guy is slow, your biryani still arrives on time. That's decomposition — split by what job it does, not by who is free.
Diagram
One big app becomes small services, each owning one business job.
Real system example
🛒 E-commerce at Amazon / Flipkart scale — one sale night, two worlds:
  1. The monolith world: a junior dev fixes a typo in the refund email template. To ship it, the entire website must be re-deployed. The deploy goes bad → checkout is down for 10 minutes during the sale → crores lost over a typo.
  2. Scaling is all-or-nothing: sale traffic is uneven — search gets 10× load, payments 8×, "edit profile" almost nothing. But a monolith scales as one block, so you must run 100 huge servers to scale everything, including parts nobody is touching.
  3. One weak corner sinks the ship: a memory leak in the Reviews code crashes the app — and takes ordering down with it.
  4. The decomposed world: the Search team deploys 5 times a day without asking anyone. Only Order and Payment pods scale to 50 during the sale. Reviews service crashes? The product page simply hides reviews — people can still buy.

This is literally Amazon's own story: they broke their monolith into services in the early 2000s and gave each small team full ownership — "you build it, you run it". That decision is what made fast, independent shipping possible at their size.

Code — same endpoint, two worlds
OrderController.java — lives in its own Spring Boot app (order-service)
// order-service — a complete, small Spring Boot app. Nothing else lives here.
@SpringBootApplication
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final OrderService orderService;

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @PostMapping
    public ResponseEntity<OrderResponse> placeOrder(@RequestBody OrderRequest request) {
        // This service ONLY knows about orders.
        // It does NOT contain user code or payment code.
        // To charge money, it will CALL payment-service over HTTP or Kafka.
        return ResponseEntity.status(HttpStatus.CREATED)
                             .body(orderService.placeOrder(request));
    }
}
How do you decide the split?
RuleMeaning in simple words
By business capabilityOne service = one thing the business does (orders, payments, shipping). Not "controller service" and "database service".
Single responsibilityIf you can't describe the service in one sentence, it's too big.
Team ownershipOne small team should be able to own, build, and deploy it end to end.
Don't start too small200 tiny services is worse than 8 sensible ones. Split more only when pain appears.
🎯 Interview one-liner: "We decompose by business capability, not by technical layer — each service owns one job, its own data, and its own deployment lifecycle."
02

Database per Service

Data

Each service gets its own database that nobody else can touch. Other services never run SQL on your tables — if they want your data, they must ask through your API or listen to your events.

Real-life analogy
🏪 Shops in a mall. Every shop keeps its own cash drawer and its own ledger. The shoe shop cannot open the coffee shop's drawer "just to check something". If the shoe shop wants to know coffee sales, it has to ask the coffee shop. Annoying? Slightly. But nobody can silently mess up someone else's accounts.
Diagram
Each service picks the database that fits its job — and hides it from everyone else.
Real system example
📸 Instagram-style app with 2 billion users — why each service picks its own database:
  1. User Service → PostgreSQL: the rules are strict — usernames must be unique, one email can't register twice, a password change must fully happen or not at all. This is "correctness first" data, exactly what a relational database with ACID transactions is built for.
  2. Post Service → Cassandra: 100M+ photos are posted per day — a firehose of writes, while reads are simple ("give me post #123"). Cassandra spreads this across hundreds of machines and keeps accepting writes even when some of them die.
  3. Feed Service → Redis: when you open the app, your feed must appear instantly. It's essentially a precomputed list of post IDs per user — a perfect fit for an in-memory store that answers in under a millisecond.

Now the horror story if all three shared one database: Friday 6 PM, the analytics team runs a heavy report query. It locks tables — and login slows down for the entire planet, even though login has nothing to do with analytics. And the User team can't rename a single column without a meeting with two other teams. With separate databases, the blast radius of any mistake stays inside one service, and each team changes its own schema freely.

Code — each service points at its own DB

⚠️ Common interview trap: "How do you JOIN across services now?" Answer: you can't, and that's on purpose. You either (a) call the other service's API, (b) keep a local read-only copy updated via events (see concept 07), or (c) do the join in an API Gateway / BFF layer. Never connect to another service's DB.
🎯 Interview one-liner: "Database per service gives loose coupling and lets each service pick the right storage engine — the trade-off is no cross-service joins and eventual consistency, which we handle with events."
03

API Gateway

Traffic

One single front door for all clients. The mobile app talks to one URL. The gateway then routes each request to the right internal service, and handles common work — auth checks, rate limiting, SSL — in one place instead of in every service.

Real-life analogy
🏨 Hotel receptionist. You don't wander the hotel looking for the laundry room, the chef, or the electrician. You call the reception. Reception knows exactly which department handles what, blocks strangers from walking into the kitchen, and won't let one guest call 500 times a minute. One number, everything sorted.
Diagram
Clients see one URL. The gateway checks, limits, and routes.
Real system example
🛵 Food delivery app (Swiggy / Zomato style) — following one tap through the gateway:
  1. You tap "Place order". The app calls one address: api.swiggy.com/api/orders. That is the only hostname the app has ever known.
  2. The gateway checks your JWT — valid? not expired? If not, the request is rejected right at the door. Order Service never even wastes a thread on it.
  3. Rate limit check — this IP has sent 900 requests in the last minute? That's a bot scraping menus. Blocked here, before it can hurt anything inside.
  4. Routing: /api/orders/** → Order Service, /api/track/** → Delivery Tracking, /api/restaurants/** → Restaurant Service. Your phone knows none of this internal map.

The payoff: when the company launches a "Dine-out" feature, it's one new route in the gateway — 50 million already-installed apps don't change at all. Teams can rename, split, or merge internal services freely; the outside URL stays stable forever. Without a gateway: 15 services each repeating the same auth code, the mobile app hard-coding 15 hostnames, and one leaked internal URL becoming a security hole.

JwtAuthFilter.java — check the token once, at the front door
@Component
public class JwtAuthFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String authHeader = exchange.getRequest().getHeaders().getFirst("Authorization");

        if (authHeader == null || !isValidJwt(authHeader.replace("Bearer ", ""))) {
            // Stop bad requests HERE. Order/Product services never even see them.
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        return chain.filter(exchange); // token OK → forward to the right service
    }
}

04

Service Discovery

Traffic

Services find each other by name, not by IP address. Instances come and go constantly (crashes, deploys, auto-scaling), so IPs change every hour. A registry keeps a live list: "order-service is currently at these 3 addresses."

Real-life analogy
📞 Saved contacts vs memorized numbers. You call "Mom", not +91-98XXXXXX21. If Mom changes her SIM, you update the contact once and every future call still works. Now imagine Mom changes her number 30 times a day — that's what pods do in Kubernetes. Hard-coding IPs is memorizing numbers; service discovery is the contacts app.
Diagram
Instances register themselves; callers ask the registry, never hard-code IPs.
Real system example
🛒 Flash sale night — how services keep finding each other while everything moves:
  1. 11:58 PM: payment-service runs 3 pods. The registry lists exactly 3 IPs, kept fresh by heartbeats every few seconds.
  2. 12:00 AM: the sale starts. Kubernetes adds 27 new pods — that's 27 IP addresses that did not exist one minute ago. Each new pod registers itself on startup. Order Service keeps calling the same name, http://payment-service, and its calls automatically spread across all 30 instances.
  3. 12:40 AM: one pod dies of an OutOfMemoryError. It stops sending heartbeats → the registry drops it within seconds → not a single order gets routed to a dead box.
  4. 2:00 AM: traffic drops, pods scale back to 3, all those IPs vanish. Nobody edited a config file all night.

Contrast with the hard-coded world: every scale-up would mean editing config files and redeploying every caller. When scale events happen 30 times a day, that's simply impossible — which is why discovery by name isn't optional in microservices.

Code — call by name with Eureka
Order service — registering & calling payment-service by name
// 1. Each service registers itself (just add spring-cloud-starter-netflix-eureka-client)
// application.yml:
//   spring.application.name: order-service
//   eureka.client.service-url.defaultZone: http://eureka:8761/eureka

@Configuration
public class RestClientConfig {

    @Bean
    @LoadBalanced   // ← the magic: resolves service NAMES via the registry
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}

@Service
public class PaymentClient {

    private final RestTemplate restTemplate;

    public PaymentClient(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    public PaymentResponse charge(PaymentRequest request) {
        // "payment-service" is a NAME, not a host.
        // Spring asks Eureka for live instances and load-balances between them.
        return restTemplate.postForObject(
                "http://payment-service/api/payments",
                request,
                PaymentResponse.class);
    }
}

05

Circuit Breaker

Resilience

When a downstream service keeps failing, stop calling it for a while and fail fast. Otherwise every request waits for a timeout, your threads get stuck waiting, and YOUR service dies too. One sick service should not infect the whole system — that's called a cascading failure.

Real-life analogy
⚡ The MCB switch in your house. When there's a short circuit, the MCB trips instantly and cuts power. It doesn't let current keep flowing until your wiring burns. After some time, you flip it back — if the fault is fixed, power flows; if not, it trips again. Same idea: trip fast, protect everything else, retry carefully.
Diagram — the 3 states
CLOSED = healthy. OPEN = tripped, fail fast. HALF-OPEN = careful probing.
Real system example
💳 Fintech checkout — the bank goes down, minute by minute:
  1. 7:00:00 PM: the bank's API starts hanging — every call now takes 30 seconds to time out.
  2. Without a breaker: within one minute, all 200 Tomcat threads of Payment Service are stuck waiting on the bank. Payment Service freezes. Order Service calls Payment, and now ITS threads get stuck too. The whole checkout is dead — one bank's outage has killed your entire platform. This chain reaction is a cascading failure.
  3. With a breaker, 7:00:40: failures cross 50% of the last 20 calls → the circuit OPENs. New calls now fail in ~1 millisecond and hit the fallback: "Payment queued — we'll confirm shortly", and the UI offers Cash on Delivery. Threads are free; browsing, ordering, refunds all still work.
  4. 7:01:10: HALF-OPEN — 3 careful test calls go through. Bank still down → snap back to OPEN. This repeats quietly in the background.
  5. 7:19 PM: the bank recovers → test calls succeed → the circuit CLOSES and queued payments retry. Nobody was paged, and customers saw a polite message instead of an endless spinner.
Code — Resilience4j in Spring Boot
BankClient.java — @CircuitBreaker with a fallback
// dependency: io.github.resilience4j:resilience4j-spring-boot3
@Service
public class BankClient {

    private final RestTemplate restTemplate;

    public BankClient(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    @CircuitBreaker(name = "bankApi", fallbackMethod = "chargeFallback")
    public PaymentResult charge(PaymentRequest request) {
        return restTemplate.postForObject(
                "https://bank.example.com/charge", request, PaymentResult.class);
    }

    // Called automatically when the circuit is OPEN or the call fails.
    // Same signature + Throwable at the end.
    private PaymentResult chargeFallback(PaymentRequest request, Throwable t) {
        // Don't lose the order — park it and retry later.
        return PaymentResult.pending("Bank is busy. Payment queued, will retry.");
    }
}
application.yml — tuning the breaker
resilience4j:
  circuitbreaker:
    instances:
      bankApi:
        sliding-window-size: 20                  # judge health over last 20 calls
        failure-rate-threshold: 50               # >50% failed → OPEN the circuit
        wait-duration-in-open-state: 30s         # stay OPEN for 30s, then HALF-OPEN
        permitted-number-of-calls-in-half-open-state: 3   # 3 test calls
        slow-call-duration-threshold: 2s         # a call slower than 2s counts as bad
        slow-call-rate-threshold: 50

OptionWhen to pick it

⚠️ Common interview trap: "Circuit breaker vs retry?" Retry helps with brief glitches (one dropped packet). A breaker protects against sustained failure. In fact, blind retries against a dying service make things worse — you triple its load. Real answer: retry with backoff, wrapped inside a circuit breaker, plus a timeout on every call.
🎯 Interview one-liner: "A circuit breaker fails fast when a dependency is unhealthy, preventing thread-pool exhaustion and cascading failures — I use Resilience4j with a fallback that degrades gracefully instead of erroring out."
06

Saga Pattern

Data

One business action often touches many services, each with its own database — so a single ACID transaction is impossible. A saga breaks the big action into small local steps. If a later step fails, you run compensating actions that undo the earlier steps.

Real-life analogy
✈️ Booking a holiday. You book a flight ✔, then a hotel ✔, then a cab — and the cab company says "no cars available". You can't "rollback" a holiday like a database. What do you do? You cancel the hotel and cancel the flight — separate undo actions, one per booking. That's a saga: a chain of steps, each with its own undo button.
Diagram — order saga with compensation
Step 3 fails → refund payment → cancel order. Every step ships with its undo.
Real system example
🛒 Order #8812 for ₹2,499 — a saga from start to refund:
  1. Step 1 — Order Service: creates order #8812 with status PENDING. This is a normal local transaction in its own database. ✔
  2. Step 2 — Payment Service: charges ₹2,499 from your card. ✔ The money has genuinely left your account.
  3. Step 3 — Inventory Service: tries to reserve the phone… but the last unit sold one second earlier to someone else. ✗ FAILS.
  4. Compensation, in reverse order: Payment Service refunds the ₹2,499 → Order Service marks #8812 CANCELLED → you see "Sorry, just sold out — your refund has been initiated."

Why it must work this way at scale: these services live on different machines, maybe different data centers, with different databases. Holding one giant lock across all of them while a user stares at a spinner would collapse under real traffic. Uber books trips the same way (assign driver → authorize payment → start trip — each step undoable). The guarantee that matters: money is never silently lost — every rupee is either charged AND delivered, or refunded by a compensating step.

Code — a simple saga orchestrator
OrderSagaOrchestrator.java — orchestration style (one brain directs the steps)
@Service
public class OrderSagaOrchestrator {

    private final OrderService orderService;
    private final PaymentClient paymentClient;
    private final InventoryClient inventoryClient;

    public OrderSagaOrchestrator(OrderService orderService,
                                 PaymentClient paymentClient,
                                 InventoryClient inventoryClient) {
        this.orderService = orderService;
        this.paymentClient = paymentClient;
        this.inventoryClient = inventoryClient;
    }

    public OrderResult placeOrder(OrderRequest request) {
        // Step 1: local transaction in order-service DB
        Order order = orderService.createPending(request);

        // Step 2: charge the customer
        PaymentResult payment;
        try {
            payment = paymentClient.charge(order);
        } catch (Exception e) {
            orderService.markCancelled(order.getId());        // undo step 1
            return OrderResult.failed("Payment failed");
        }

        // Step 3: reserve stock
        try {
            inventoryClient.reserve(order);
        } catch (Exception e) {
            paymentClient.refund(payment.getTxnId());          // undo step 2
            orderService.markCancelled(order.getId());         // undo step 1
            return OrderResult.failed("Out of stock — amount refunded");
        }

        orderService.markConfirmed(order.getId());
        return OrderResult.success(order.getId());
    }
}
// In production the steps usually run via Kafka commands/events (async),
// and the orchestrator persists saga state so it survives a crash mid-way.
Two flavors of saga
FlavorHow it works
Orchestration easier to reason aboutOne central "conductor" (like the code above) tells each service what to do next and triggers undo steps. Tools: Temporal, Camunda, Axon, or a custom Spring service + Kafka.
ChoreographyNo conductor. Each service reacts to the previous event: Order emits OrderCreated → Payment listens, charges, emits PaymentCompleted → Inventory listens… Simple for 2–3 steps, hard to follow for many.
⚠️ Common interview trap: "Why not just use a distributed transaction (2PC)?" Because 2PC needs every participant to lock resources and wait for a coordinator — it's slow, blocks under failure, and most modern stores (Kafka, Cassandra, DynamoDB) don't support it anyway. Sagas trade instant consistency for availability: the system is briefly "in between", but always ends up correct.
🎯 Interview one-liner: "A saga replaces one impossible distributed transaction with a sequence of local transactions plus compensating actions — I prefer orchestration for complex flows because the whole state machine lives in one visible place."
07

Event-Driven Communication

Messaging

Instead of Service A directly calling B, C, and D, it just announces "this happened" on a message broker (like Kafka). Whoever cares, listens. A doesn't know or care who is listening — services get decoupled.

Real-life analogy
🔔 YouTube channel notifications. When a creator uploads a video, they don't personally call each of their 2 million subscribers. They just publish. Everyone who hit the bell gets notified automatically. New subscriber joins? The creator changes nothing. One subscriber's phone is off? The video still exists, waiting for them. Publisher and subscribers never talk directly.
Diagram
One event, many independent listeners. Adding a listener costs the publisher nothing.
Real system example
📸 Instagram — the two seconds after you tap Share:
  1. 9:00:00.000 PM: you tap Share. Post Service saves the photo reference to Cassandra and publishes ONE PostCreated event to Kafka. Total: ~50ms. You see "Posted ✔" and go back to scrolling — your part is completely over.
  2. Over the next second, independently and in parallel: Feed Service pushes the post into your followers' timelines · Notification Service pings your close friends · Media Service generates 5 thumbnail sizes · Analytics counts it · an ML service tags the content. Five teams, zero coordination between them.
  3. The failure test: Notification Service is down for 10 minutes? Its events simply wait in Kafka and get processed when it returns. Friends get pinged a bit late — but your upload never noticed anything.

Now imagine the REST version: posting a photo = 5 synchronous calls. The slowest one holds your spinner hostage, any one failing fails your upload, and adding the ML service means editing and redeploying Post Service. With events, the ML team just subscribed to the topic — the Post Service team didn't even know it happened.

Code — Spring Kafka producer & consumer
Producer (order-service) and Consumer (notification-service)
// ---------- order-service: publish and move on ----------
@Service
public class OrderEventPublisher {

    private final KafkaTemplate<String, OrderPlacedEvent> kafkaTemplate;

    public OrderEventPublisher(KafkaTemplate<String, OrderPlacedEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void publishOrderPlaced(Order order) {
        OrderPlacedEvent event = new OrderPlacedEvent(
                order.getId(), order.getUserId(), order.getTotalAmount());
        // key = orderId → all events of one order land in the same partition, in order
        kafkaTemplate.send("order-events", order.getId().toString(), event);
    }
}

// ---------- notification-service: a completely separate app ----------
@Component
public class OrderEventsListener {

    @KafkaListener(topics = "order-events", groupId = "notification-service")
    public void onOrderPlaced(OrderPlacedEvent event) {
        // Order Service has no idea this class exists. That's the point.
        emailService.sendOrderConfirmation(event.userId(), event.orderId());
    }
}
Sync call vs event — when to use which?
Use a direct REST call when…Use an event when…
You need the answer NOW to continue (e.g., "is this card valid?").It's a side effect that can happen a moment later (email, feed update, analytics).
The caller must know if it failed.Many services care about the same fact.
Simple request/response between two services.You want to add listeners later without touching the publisher.
🎯 Interview one-liner: "Events decouple producers from consumers in both time and knowledge — the publisher survives consumer outages, and new consumers can be added with zero upstream changes. I use Kafka with the entity ID as the partition key to keep per-entity ordering."
08

Transactional Outbox

Messaging

The dual-write problem: you save to your database AND publish to Kafka — two separate systems, so one can succeed while the other fails. The fix: write the event into an outbox table in the same DB transaction as your data. A separate process then reliably ships outbox rows to Kafka.

Real-life analogy
📬 The office outbox tray. You finish a contract and, in the same sitting, place a copy in the "outgoing mail" tray on your desk. You don't run to the post office yourself — the office peon collects the tray every 15 minutes and posts everything. If the peon is on chai break, letters just wait safely in the tray. Nothing gets lost, because "signing the contract" and "putting it in the tray" happen together as one action.
Diagram
Data + event committed atomically in one DB; a poller ships events to Kafka afterwards.
Real system example
💰 Wallet transfer at 9:15 PM — the crash that decides everything:
  1. The naive code: line 1 db.save(debit) succeeds — ₹5,000 leaves your wallet. Then the pod is killed (out of memory) before line 2 kafka.send(event) runs.
  2. The damage: your wallet shows −₹5,000, but the Ledger never recorded it and the receiver was never notified. The money is in limbo, and a support human now reconciles it by hand. At even a 0.01% failure rate on millions of daily transfers, that's hundreds of angry tickets every single day.
  3. With the outbox: the debit row and the event row commit in ONE database transaction at 9:15:00. Pod dies right after? Fine — a fresh pod's poller finds the PENDING outbox row at 9:15:02 and publishes it. Nothing is lost.
  4. The other crash: poller publishes to Kafka but dies before marking the row SENT? The event goes out twice — and consumers de-duplicate by event ID (that's idempotency, concept 14).

"At-least-once delivery + idempotent consumers" is exactly how real payment infrastructure — Stripe, PayPal, UPI switches — guarantees an event is never lost. Duplicates are acceptable and handled; disappearance is not.

Code — write + outbox in one transaction, poller publishes
OrderService.java + OutboxPublisher.java
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final OutboxRepository outboxRepository;
    private final ObjectMapper objectMapper;

    // ONE local ACID transaction — both rows, or none. No dual-write problem.
    @Transactional
    public Order placeOrder(OrderRequest request) throws JsonProcessingException {
        Order order = orderRepository.save(Order.confirmed(request));

        OutboxEvent event = new OutboxEvent(
                "Order",                                   // aggregate type
                order.getId().toString(),                  // aggregate id
                "OrderPlaced",                             // event type
                objectMapper.writeValueAsString(order));   // payload (JSON)
        outboxRepository.save(event);                      // same DB, same txn

        return order;
    }
}

@Component
public class OutboxPublisher {

    private final OutboxRepository outboxRepository;
    private final KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedDelay = 2000)   // every 2s: pick up pending letters from the tray
    public void publishPendingEvents() {
        List<OutboxEvent> pending = outboxRepository.findTop100ByStatusOrderByCreatedAt("PENDING");

        for (OutboxEvent event : pending) {
            kafkaTemplate.send("order-events", event.getAggregateId(), event.getPayload())
                         .whenComplete((result, ex) -> {
                             if (ex == null) {
                                 event.setStatus("SENT");   // mark done only after Kafka ack
                                 outboxRepository.save(event);
                             }
                             // on failure: stays PENDING → retried next cycle
                         });
        }
    }
}
// Consumers must be IDEMPOTENT (handle duplicates) — this design is
// "at-least-once": an event may be sent twice, but never zero times.
Options for shipping the outbox
OptionWhen to pick it
Polling publisher simplestA @Scheduled job like above. Easy, works everywhere. Small delay (poll interval) and some DB load.
CDC with DebeziumDebezium tails the DB's transaction log (WAL/binlog) and streams outbox rows to Kafka — near real-time, no polling queries, the production-grade choice.
🎯 Interview one-liner: "The outbox pattern solves the dual-write problem: the state change and the event commit atomically in one local transaction, and a poller or Debezium CDC relays events to Kafka with at-least-once delivery — consumers handle dedup via idempotency."

Centralized Logging & Distributed Tracing

Observability

One user click can travel through 6 services. When it's slow or broken, you need to see the whole journey as one story. Tracing gives every request a trace ID that travels with it across services; centralized logging puts all logs in one searchable place, tagged with that ID.

Real-life analogy
📦 Courier tracking number. Your parcel goes Pune hub → Mumbai hub → flight → Delhi hub → delivery van. One tracking number follows it everywhere, and the app shows every checkpoint with timestamps: you instantly see it's been stuck at Mumbai for 2 days. Without the number, you'd be phoning five different warehouses asking "have you seen a brown box?" — that's grepping logs on 6 servers.
Diagram
Same trace ID at every hop → a waterfall view shows exactly where time went.
Real system example
💼 SaaS company, Monday morning — solving "checkout is slow" in six minutes:
  1. 10:00 AM: a ticket arrives — "checkout takes 4+ seconds for some users." Six services are involved. The old way: SSH into six boxes, grep logs by timestamps, and guess. That's hours, maybe days.
  2. 10:02: an engineer opens Jaeger, filters service = gateway, duration > 3s, and clicks one trace. The waterfall reads like a story: gateway 5ms → order 30ms → payment 40ms → bank API 3.8 seconds. Found it.
  3. 10:04: she copies traceId 7f3a into Kibana. Every log line from all four services appears in exact order, as one story — payment's logs show the bank returning HTTP 503 and retrying three times.
  4. 10:06: root cause posted in the ticket: the bank's certificate expired. Total time: six minutes, zero SSH sessions.

At Netflix or Uber scale — hundreds of services, millions of requests per minute — this isn't a luxury. Without a trace ID stitching the hops together, production debugging is archaeology.

Containerization & Orchestration

Infra

A container (Docker) packs your service + JDK + configs into one standard box that runs the same everywhere. An orchestrator (Kubernetes) then manages hundreds of these boxes: starts them, restarts crashed ones, spreads traffic, and scales them up and down automatically.

Real-life analogy
🚢 Shipping containers. Before standard containers, loading a ship was chaos — sacks, barrels, crates, each handled differently. Standard steel containers changed everything: the crane doesn't care what's inside; every container fits every ship, truck, and train. Docker is that standard box for software. And Kubernetes is the port authority — deciding which ship carries which containers, replacing damaged ones, adding more ships during festival season.
Diagram
K8s heals crashed pods and scales the hot service automatically.
Real system example
🛒 Flash sale night — Kubernetes runs the show, hour by hour:
  1. 11:50 PM: order-service runs 3 pods at 30% CPU. The autoscaler (HPA) checks metrics every 15 seconds and sees nothing to do.
  2. 12:00 AM: the sale starts. CPU hits 85% → HPA scales 3 → 10 → 22 → 30 pods over about two minutes. The scheduler places new pods on nodes with free capacity, and readiness probes hold traffic back until each Spring Boot app has fully started — so no request ever hits a half-booted pod.
  3. 12:34 AM: one pod throws OutOfMemoryError and hangs. Its liveness probe fails → Kubernetes kills and replaces it automatically. The other 29 pods absorb the traffic; users never notice.
  4. 2:00 AM: CPU falls to 20% → gradual scale-down back to 3 pods. The cloud bill only covers what was actually used.

All night, user-service quietly stayed at 3 pods — that's the payoff of concept 01: you scale only what's hot. And no human SSH-ed into anything. Netflix runs thousands of services exactly this way.

SQL vs NoSQL — Picking the Right Database

Data

There is no "best" database. Each type is shaped for a different kind of data and a different way of asking questions. Since every service owns its own database (concept 02), every service can pick the type that fits its job. Using many database types together is called polyglot persistence.

Real-life analogy
🏠 Storage at home. You don't keep everything in one steel almirah. Clothes go in the wardrobe, shoes in the shoe rack, spices in small labelled boxes, milk in the fridge. Each storage is shaped for what it holds and how often you grab it. Forcing everything into the almirah "because it's the biggest one" — milk included — is what using one database for everything looks like.
Diagram — match the data shape to the database
Five questions, five shapes, five different databases — all in one system.
Real system example
🛒 One e-commerce company, five databases — each doing what it's best at:
  1. Orders & payments → PostgreSQL/MySQL: money data has strict rules — an order total must equal the sum of its items, a payment can't half-happen. SQL transactions (ACID) enforce this automatically. This is why banks still run their core ledgers on relational databases.
  2. Product catalog → MongoDB (document): a TV has screen size and refresh rate; a shirt has size and fabric; a book has author and page count. In SQL you'd need dozens of mostly-empty columns, or a painful table change every time a new category launches. In MongoDB, each product is one flexible JSON document — new category, new fields, zero migrations.
  3. Cart & sessions → DynamoDB / Redis (key-value): the question is always identical — "give me THE cart of user 831". No joins, no searching, just grab-by-key, millions of times a minute. Amazon's shopping cart famously runs on this idea — its early cart problems are literally why Amazon invented DynamoDB.
  4. Viewing history / activity feed → Cassandra (wide-column): Netflix writes billions of "user watched X at time Y" rows per day into Cassandra. Insane write volume, reads always by user ID, data spread across hundreds of machines — and it keeps working when some of those machines die.
  5. "People also bought" → Neo4j (graph): "friends of friends who bought this item" is a nightmare of self-joins in SQL, but a natural two-step walk in a graph database. It's the same shape as LinkedIn's "People you may know".
Code — three services, three storage styles
Same company: JPA entity vs Mongo document vs Redis key-value
// ---------- order-service: money → JPA + PostgreSQL (strict, ACID) ----------
@Entity
@Table(name = "orders")
public class Order {
    @Id @GeneratedValue
    private Long id;
    private Long userId;
    private BigDecimal totalAmount;   // money → SQL + transactions. Non-negotiable.
    private String status;
}

// ---------- product-service: catalog → MongoDB (flexible documents) ----------
@Document(collection = "products")
public class Product {
    @Id
    private String id;
    private String name;
    private Map<String, Object> attributes;  // TV: screenSize; shirt: fabric.
                                             // New category tomorrow? No migration.
}

// ---------- cart-service: grab-by-key → Redis ----------
@Service
public class CartService {

    private final RedisTemplate<String, Cart> redis;

    public CartService(RedisTemplate<String, Cart> redis) { this.redis = redis; }

    public Cart getCart(String userId) {
        return redis.opsForValue().get("cart:" + userId);        // one lookup, <1ms
    }

    public void saveCart(String userId, Cart cart) {
        redis.opsForValue().set("cart:" + userId, cart, Duration.ofDays(7));
    }
}
Cheat sheet
TypeAsk yourselfReal user at scale
SQL (Postgres, MySQL)"Do I need strict correctness and joins?"Every bank ledger; order systems everywhere
Key-Value (Redis, DynamoDB)"Is it always fetch-by-one-key?"Amazon cart, session stores
Document (MongoDB)"Does every record have different fields?"Product catalogs, content/CMS data
Wide-column (Cassandra)"Huge nonstop writes, reads by partition key?"Netflix viewing history, Instagram feed storage

⚠️ Common interview trap: "NoSQL is faster than SQL, right?" No — NoSQL is not "SQL but faster". You give up joins and cross-row transactions to gain scale for one specific access pattern. Golden rule: in NoSQL you model around your queries, not your entities — in Cassandra you literally design one table per query.
🎯 Interview one-liner: "I pick storage per access pattern — ACID and joins → relational; grab-by-key → key-value; flexible schema → document; massive partitioned writes → wide-column; relationship traversal → graph. Database-per-service is what makes mixing them safe."

Caching with Redis

Performance

Keep frequently-read data in memory, right next door, so most reads never touch the database at all. RAM answers in under a millisecond; a DB query takes tens of milliseconds. Even more important: the cache shields the database from millions of identical queries.

Real-life analogy
🚰 Water bottle on your desk. You don't walk to the kitchen for every sip — you fill a bottle once and drink from it all morning. The bottle can be slightly less fresh than the tap (that's staleness), and it eventually runs empty and needs a refill (that's TTL expiry). One kitchen trip replaced fifty. That's a cache.
Diagram — cache-aside pattern
Check cache → on miss, read DB once → store with TTL. The DB sees ~1 query per key per minute.
Real system example
📸 Instagram — a celebrity posts:
  1. 7:00 PM: Virat Kohli posts a photo. Within 10 minutes, 5 million people open his profile. That is the SAME profile row and the SAME recent posts being requested 5 million times.
  2. Without a cache: 5 million identical SQL queries hammer PostgreSQL. Connections max out, every other page on the site slows down, and the DB can fall over. You've DDoS-ed yourself with your own popularity.
  3. With cache-aside: the very first request misses → reads the DB once (50ms) → stores the result in Redis with a 60-second TTL. The next ~8,000 requests per second are answered from memory in under a millisecond. The database now sees roughly ONE query per minute for that profile instead of millions.
  4. He edits his bio: the service updates the DB and deletes the cache key. The next read misses and repopulates with fresh data. Worst case, a few people see the old bio for a few seconds — totally fine for a bio, never acceptable for a wallet balance (don't cache money blindly).
Code — Spring's cache abstraction over Redis
ProductService.java — @Cacheable + @CacheEvict
@Service
public class ProductService {

    private final ProductRepository productRepository;

    public ProductService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    // 1st call: runs the method, stores the result in Redis as "products::42".
    // Every call after that (until TTL expires) is served straight from Redis —
    // this method body doesn't even execute.
    @Cacheable(cacheNames = "products", key = "#productId")
    public Product getProduct(long productId) {
        return productRepository.findById(productId)
                .orElseThrow(() -> new ProductNotFoundException(productId));
    }

    // On update: fix the DB, then DELETE the stale cache entry.
    // (Deleting is safer than trying to update the cache in place.)
    @CacheEvict(cacheNames = "products", key = "#product.getId()")
    public Product updateProduct(Product product) {
        return productRepository.save(product);
    }
}
application.yml — Redis as the cache store, with TTL
spring:
  cache:
    type: redis
    redis:
      time-to-live: 60s     # every entry self-destructs after 60s → self-healing staleness
  data:
    redis:
      host: redis
      port: 6379
Options
OptionWhen to pick it
Redis default choiceShared cache all service instances see; rich data types; also does locks, counters, queues.
Caffeine (in-process)Ultra-hot data cached inside the JVM itself (L1), often layered in front of Redis (L2).
MemcachedDead-simple string cache; fine, but Redis does more and is the ecosystem standard now.
CDN (CloudFront)Caching for images/static files at the network edge, close to users — a cache too, just geographic.
⚠️ Common interview trap: "What's hard about caching?" Invalidation. Rules of thumb: always set a TTL (self-healing), delete the key on update instead of updating it, and never cache data where staleness costs money (wallet balance, live stock count). Bonus term to drop: cache stampede — a hot key expires and thousands of requests hit the DB at the same instant; fixed with a small lock or randomized TTL jitter.
🎯 Interview one-liner: "Cache-aside with TTL: read cache → on miss read DB and populate → invalidate by deleting on write. It turns five million identical reads into one DB query per TTL window — the cheapest 10× win in system design."

CQRS — Separate Read & Write Models

Architecture

Command Query Responsibility Segregation: writes and reads take different paths. Writes (Commands) go into a strict, normalized model. Reads (Queries) are served from a separate model that is pre-flattened for viewing — kept up to date by events. Useful when reads and writes have very different shapes or very different scale.

Real-life analogy
🍽️ The food-court order screen. You place your order at the counter — the cashier validates it, takes payment, prints a kitchen ticket. That's the write side: careful, strict, one at a time. But the big "PREPARING → READY" screen the whole crowd watches is a separate display, updated as the kitchen makes progress. Two hundred people read the screen without ever touching the kitchen's ticket book. Writes and reads: separated.
Diagram
Writes stay strict on the left; events build a ready-to-show view; reads hit only the view.
Real system example
🛒 Amazon-style "My Orders" page:
  1. The page shows order details + product name and image + live delivery status. That data is owned by three different services: Order, Product, and Shipping.
  2. The naive way: every single page view makes 3 network calls and stitches the results together. Millions of views per hour × 3 calls each = enormous load, a slow page, and if Product Service is down, your orders page breaks — even though your orders are fine.
  3. The CQRS way: whenever OrderPlaced, OrderShipped, or ProductUpdated events fire, a small projector updates ONE flattened "order history" document per order in MongoDB or Elasticsearch — name, image URL, status, ETA, all baked in. The page is now a single query taking ~10ms, no matter how many services own the underlying data.
  4. Writes never got looser: placing an order still goes through the strict PostgreSQL write model with full validation. Only the viewing got its own fast lane. Instagram's feed (concept 07) is this same idea at extreme scale — write the post once, precompute the read side for millions of followers.
Code — projector builds the view, reads hit only the view
OrderHistoryProjector.java + OrderHistoryController.java
// ---------- read-side builder: listens to events, flattens data ----------
@Component
public class OrderHistoryProjector {

    private final OrderHistoryViewRepository viewRepository;

    public OrderHistoryProjector(OrderHistoryViewRepository viewRepository) {
        this.viewRepository = viewRepository;
    }

    @KafkaListener(topics = {"order-events", "shipping-events"},
                   groupId = "order-history-projector")
    public void on(OrderEvent event) {
        // Merge facts from many services into ONE ready-to-show document
        OrderHistoryView view = viewRepository.findByOrderId(event.orderId())
                                              .orElseGet(OrderHistoryView::new);
        view.apply(event);           // sets status, product name, image URL, ETA...
        viewRepository.save(view);   // stored flattened in MongoDB
    }
}

// ---------- read side: one cheap query. No joins. No service calls. ----------
@RestController
@RequestMapping("/api/order-history")
public class OrderHistoryController {

    private final OrderHistoryViewRepository viewRepository;

    public OrderHistoryController(OrderHistoryViewRepository viewRepository) {
        this.viewRepository = viewRepository;
    }

    @GetMapping("/{userId}")
    public List<OrderHistoryView> history(@PathVariable String userId) {
        return viewRepository.findByUserIdOrderByCreatedAtDesc(userId);  // ~10ms
    }
}
⚠️ Common interview trap: "Why not use CQRS everywhere?" Because it's a complexity tax: two models, a projector, and eventual consistency (the view lags writes by milliseconds to seconds — a user might place an order and not see it in history for a second). For a simple CRUD service it's over-engineering. Reach for CQRS when reads and writes differ wildly in shape or scale.
🎯 Interview one-liner: "CQRS keeps the write model strict and builds purpose-shaped read models from events — reads become one cheap query instead of runtime joins across services, at the cost of eventual consistency on the view."

Idempotency — Safe Retries

Resilience

An operation is idempotent if doing it twice has the same effect as doing it once. In distributed systems, timeouts, retries, and Kafka's at-least-once delivery mean duplicates WILL happen — idempotency is what makes them harmless.

Real-life analogy
🛗 The lift button. Pressing it five times doesn't summon five lifts — the first press registers, the rest are absorbed. Now compare an ATM's cash button: press twice, get paid twice. You want every API you build to behave like the lift button, never like the ATM button.
Diagram
Same key twice → second attempt gets the saved answer, not a second charge.
Real system example
💳 Fintech — the double-payment that never happened:
  1. 9:41 PM: you tap Pay ₹999. The bank actually charges the card — but the response gets lost in a network blip. Your app only sees a timeout. Did the payment happen? Nobody on the client side knows.
  2. The app retries automatically (which is the right thing to do!) — sending the exact same request with the same Idempotency-Key: abc-123 header it generated the first time.
  3. Server side: Payment Service runs SETNX idem:abc-123 in Redis. The key already exists from attempt #1 — so it does NOT charge again. It looks up and returns the stored response of the original attempt: "Payment successful."
  4. Result: one charge, one happy customer. This is literally how Stripe's public API works — every POST accepts an Idempotency-Key. Without it, "should I retry a payment timeout?" has no safe answer: don't retry and the customer may never be charged; retry and they may be charged twice.
Same idea on the Kafka side (remember concept 08 delivers at-least-once): consumers record processed event IDs — a unique constraint or dedup table — and silently skip repeats.
Code — idempotency key with Redis SETNX
PaymentService.java — claim the key, charge once, replay duplicates
@Service
public class PaymentService {

    private final StringRedisTemplate redis;

    public PaymentService(StringRedisTemplate redis) { this.redis = redis; }

    public PaymentResponse pay(String idempotencyKey, PaymentRequest request) {

        // SETNX = "set only if not exists" → atomic claim of this key
        Boolean firstTime = redis.opsForValue()
                .setIfAbsent("idem:" + idempotencyKey, "IN_PROGRESS", Duration.ofHours(24));

        if (Boolean.FALSE.equals(firstTime)) {
            // Duplicate (a retry). Return the ORIGINAL result — do NOT charge again.
            String saved = redis.opsForValue().get("result:" + idempotencyKey);
            return PaymentResponse.fromJson(saved);
        }

        PaymentResponse response = chargeBank(request);   // happens exactly once
        redis.opsForValue().set("result:" + idempotencyKey,
                                response.toJson(), Duration.ofHours(24));
        return response;
    }
}

// Controller: pay(@RequestHeader("Idempotency-Key") String key, @RequestBody ...)
// Belt-and-braces: also put a UNIQUE constraint on payments.idempotency_key
// in the DB — it catches duplicates even if Redis is wiped.
Strategies
StrategyWhere it fits
Naturally idempotent verbsPUT ("set status to SHIPPED") and DELETE are safe to repeat by design. POST ("add ₹100") is not.
Idempotency-Key header for POSTsClient generates a UUID per logical action; server stores key → response (Stripe's model).

⚠️ Common interview trap: "Can't Kafka just give exactly-once?" Across system boundaries, exactly-once delivery is practically a myth. Real systems do at-least-once delivery + idempotent processing — which from the outside looks exactly-once. Say that sentence in an interview and watch them nod.
🎯 Interview one-liner: "Retries and at-least-once delivery make duplicates a question of when, not if — so every money-moving endpoint takes an idempotency key, claims it atomically (SETNX or unique constraint), and replays the stored response for repeats."

Strangler Fig — Monolith to Microservices

Architecture

You almost never rewrite a monolith in one go. Instead: put a gateway in front, carve out one capability at a time into a new service, route just that traffic to it, and repeat — until the monolith is an empty shell you switch off. Named after the strangler fig tree, which grows around a host tree and slowly replaces it.

Real-life analogy
🏠 Renovating room by room — while living in the house. You don't demolish the whole house and sleep on the street for two years (that's the big-bang rewrite). You redo the kitchen first, keep living normally, then the bathroom, then a bedroom. One day you look around and the whole house is new — and you never moved out.
Diagram
The gateway peels traffic away route by route; the monolith shrinks until it's gone.
Real system example
🏦 A 10-year-old e-commerce/banking monolith, migrated safely:
  1. Month 0: put an API gateway in front. Change nothing else — every route still goes to the monolith. Users notice nothing, but you now own the front door, which is what makes everything after this possible.
  2. Months 1–3: pick one capability with clean edges — say Notifications. Build notification-service, then flip a single gateway route: /api/notifications/** → new service. Everything else still hits the monolith. Found a bug? Flip the route back in seconds — your rollback plan is one line of config.
  3. Months 4–8: next, Payments — riskier, so run it as a canary: 10% of payment traffic to the new service while watching error rates and latency dashboards, then 50%, then 100%. Real users validated it gradually; no leap of faith.
  4. Months 9–24: repeat, capability by capability. The monolith shrinks like the host tree inside the fig. One quiet Tuesday, the last route flips, the monolith is switched off — and no user ever saw a "we are migrating" banner.
This is how the giants actually did it — Amazon and Netflix each spent years strangling their monoliths piece by piece while shipping features the whole time. The graveyard of "we'll rewrite everything from scratch in 18 months" projects is far larger.
