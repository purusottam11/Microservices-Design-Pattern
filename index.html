Why companies leave monoliths behind, one bottleneck at a time

Every pattern here exists because a monolith hit a specific wall in production. For each of the 15 concepts: the real problem monoliths run into, why the monolith architecture structurally can't fix it, how splitting into services solves it, how Amazon/Flipkart/Uber/Netflix/Tinder/Google actually did it, and the exact tech (Kafka, Redis, Kubernetes...) with the reason it was chosen for that job.

The one mental model behind all 15 patterns

A monolith is one codebase, one build, one deploy, one database, one runtime. That's fantastic on day one — one repo to clone, one place to debug, transactions just work. The problem was never the code quality. The problem is that one of everything means every axis — scaling, deploying, failing, hiring — is shared across the entire business, whether it wants to be or not.

Microservices don't fix bugs or add features. They exist to give each business capability its own axis: its own deploy schedule, its own scaling knob, its own failure boundary, its own team, its own database engine. Every pattern below is really the same move applied to a different axis — that repetition is the mental model.

One honest catch: splitting the monolith isn't free. It trades old bottlenecks for new problems — services now need a way to find each other, which is what an API Gateway and Service Discovery solve. One sick service (say, a slow bank integration) shouldn't be able to freeze every other service waiting on it — that's what a Circuit Breaker prevents. Data that used to live in one database now has to stay consistent across several — that's what the Saga Pattern and Transactional Outbox solve. And a request that now hops across 6 services needs a way to be followed end to end — that's Distributed Tracing. Most of this guide, after the first couple of ideas, is really the set of answers to the problems splitting itself creates.

Every concept below is this same split, applied to deploys, scaling, failure, data, or ownership.
01

Service Decomposition

Design

Break one application into small services, each owning one business capability — its own code, its own team, its own deploy. This is the root decision every other pattern in this guide builds on.

🏚️ The Monolith Reality
A typical growth story: an e-commerce app starts as one Spring Boot WAR — UserController, OrderController, PaymentController, InventoryController, all in one repo, one build, one running process. Eighteen months and 40 engineers later, a junior dev fixes a one-line typo in the refund email template. To ship it, the entire application has to rebuild, redeploy, and restart — because that's the only unit of deployment that exists.
🚧 Why the Monolith Can't Fix This
It's not a process problem — it's structural. A monolith is one build artifact, one running process, one shared heap:
  • Deploy is all-or-nothing: there's no way to ship only the email-template fix — the deploy pipeline doesn't know your change was 3 lines out of 400,000.
  • Scale is all-or-nothing: Search gets 10× the load Payments does on sale day, but they run in the same process — you scale the whole monolith just to give Search more headroom.
  • Failure is shared: a memory leak in the Reviews module eventually OOMs the whole JVM — and Ordering, which has nothing to do with Reviews, goes down with it.
  • Ownership is shared: 40 engineers committing to one repo means merge conflicts, one team's bad migration blocking everyone's release train, and nobody able to say "this is fully mine."
🔀 How Microservices Solve It
Split by business capability, not technical layer. Order Service, Payment Service, and Inventory Service become separate deployable units — separate repos or at least separate build/deploy pipelines, separate runtimes, separate teams. The Search team now ships 5 times a day without asking Payments. A memory leak in Reviews takes down only Reviews. Scaling decisions become per-service knobs instead of one global one.
One shared process becomes three independent ones — each with its own deploy, scale, and blast radius.
🌍 Proven at Scale
  • Amazon ran one large C++ monolith through the late 1990s. Order volume grew, and the same story repeated: unrelated teams blocked each other's releases, and one bad change could take down checkout. Amazon's leadership eventually mandated that all teams expose their functionality only through well-defined APIs, effectively forcing the monolith apart into service-owning teams — the origin of the "you build it, you run it" culture that's now industry-wide.
  • Uber started as a single Python monolith dispatching rides in one city. As they expanded city by city, one city's edge-case code (say, surge pricing rules unique to São Paulo) risked breaking rides in Chicago on every deploy. They decomposed into services by capability — pricing, dispatch, maps, payments — specifically so a bug in one domain couldn't take down rides globally.

OrderController.java — lives in its own Spring Boot app (order-service)
// order-service — a complete, small Spring Boot app. Nothing else lives here.
@SpringBootApplication
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final OrderService orderService;

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @PostMapping
    public ResponseEntity<OrderResponse> placeOrder(@RequestBody OrderRequest request) {
        // This service ONLY knows about orders.
        // To charge money, it CALLS payment-service over HTTP or Kafka —
        // it does not contain payment code in the same process.
        return ResponseEntity.status(HttpStatus.CREATED)
                             .body(orderService.placeOrder(request));
    }
}

02

Database per Service

Data

Each service gets its own database that nobody else touches. Other services never run SQL on your tables — they ask through your API or listen to your events.

🏚️ The Monolith Reality
One database, every team's fingerprints on it: the monolith has a single PostgreSQL instance. The Analytics team runs a heavy end-of-day report query Friday at 6 PM — it takes table locks, and suddenly login slows down for the entire platform, even though login has nothing to do with analytics. Meanwhile, the User team wants to rename one column, but three other teams' queries touch that table, so it takes a scheduling meeting and a migration window at 2 AM.
🚧 Why the Monolith Can't Fix This
One shared schema means one shared blast radius and one shared bottleneck, structurally:
  • Lock contention is shared: any heavy query — a report, a backfill, a bad index — can degrade every other feature reading the same tables, because they're the same physical database.
  • Schema changes require cross-team consensus: you can't evolve your table's shape without checking every other team's queries against it first.
  • One storage engine has to fit every access pattern: a strict relational DB is right for orders and payments, but wrong for a fast-growing photo feed or a session store — yet the monolith forces everyone onto the same engine.
🔀 How Microservices Solve It
Give each service its own database, and let it pick the engine that fits its job. A report query in Analytics's own database can never lock User Service's login table — they're physically different systems. Schema changes become a one-team decision. And each service can choose Postgres, Cassandra, MongoDB, or Redis based on its actual read/write shape, not a company-wide default.
Each service picks the database that fits its job — and hides it from everyone else.
🌍 Proven at Scale
  • Instagram-scale photo apps split exactly this way: User Service on PostgreSQL (strict uniqueness rules, ACID for account changes), Post Service on Cassandra (100M+ photo writes/day, simple reads-by-ID), Feed Service on Redis (precomputed timelines, sub-millisecond reads). One shared database could never serve all three access patterns well at once.
  • Netflix stores viewing history — billions of "user watched X at time Y" events per day — in Cassandra specifically because it accepts nonstop writes across hundreds of nodes and keeps working when some nodes fail; that write volume would choke a single relational instance shared with, say, billing.
  • Flipkart's product catalog spans wildly different product types — a phone has RAM and screen size, a saree has fabric and length — which is exactly the flexible-schema problem a document store like MongoDB solves; forcing it into the same relational schema as Orders (which genuinely needs strict transactions) would mean dozens of mostly-empty columns and constant migrations.

⚠️ Common interview trap: "How do you JOIN across services now?" You can't, on purpose. You either (a) call the other service's API, (b) keep a local read-only copy that stays updated via published events — e.g. Order Service caching a product's name and image locally, refreshed whenever Product Service emits a ProductUpdated event — or (c) join in an API Gateway/BFF layer. Never connect to another service's database directly.
🎯 Interview one-liner: "Database per service removes the shared lock contention and shared schema of a monolith, and lets each service pick the storage engine that matches its access pattern — the trade-off is no cross-service joins and eventual consistency, handled with events."
03

API Gateway

Traffic

One single front door for all clients. The gateway routes each request to the right internal service and handles cross-cutting work — auth, rate limiting, SSL — once, instead of in every service.

🏚️ The Monolith Reality
The monolith had this for free: one host, one port, one auth filter, done. The moment you decompose into Order, Product, and User services — each its own deployable app on its own host — that simplicity disappears — the naive fix is for the mobile app to call all three hosts directly. Now the app hard-codes three internal hostnames, each service duplicates its own JWT-checking code, and there's no single place to stop an abusive client hammering your servers.
🚧 Why "Just Call Each Service Directly" Doesn't Scale
Direct client-to-service calls recreate monolith-style coupling, just distributed across the network instead of inside one process:
  • Clients become coupled to internal topology: rename or split a service, and every client — mobile, web, partner integrations — breaks.
  • Cross-cutting logic gets duplicated N times: auth, rate limiting, and request logging get copy-pasted into every one of 15+ services, drifting out of sync as each team patches it separately.
  • No single choke point for abuse: without one front door, there's nowhere to enforce "this IP gets 100 requests/min" across the whole API surface.
🔀 How Microservices Solve It
Put one gateway in front of everything. Clients only ever know one hostname. The gateway verifies the JWT once, enforces rate limits once, and routes by path to the right internal service — which can be renamed, split, or merged behind the scenes without any client ever noticing.
Clients see one URL. The gateway checks, limits, and routes — internal topology stays hidden.
🌍 Proven at Scale
  • Swiggy/Zomato-style food delivery apps: the app calls one address, e.g. api.swiggy.com. The gateway checks the JWT, rejects bots hammering the menu endpoint, and routes /orders/**, /track/**, and /restaurants/** internally — the phone never learns this internal map. Launching a new "Dine-out" feature is one new gateway route; 50 million installed apps don't change.
  • Uber runs thousands of internal services (pricing, dispatch, maps, payments) behind a small number of edge gateway layers — the rider and driver apps only ever talk to Uber's public edge, never to internal service hostnames, which is what lets Uber refactor internal services constantly without ever shipping an app update for it.

JwtAuthFilter.java — check the token once, at the front door
@Component
public class JwtAuthFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String authHeader = exchange.getRequest().getHeaders().getFirst("Authorization");

        if (authHeader == null || !isValidJwt(authHeader.replace("Bearer ", ""))) {
            // Stop bad requests HERE. Order/Product services never even see them.
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        return chain.filter(exchange);
    }
}

04

Service Discovery

Traffic

Services find each other by name, not by IP address. Instances come and go constantly — a registry keeps a live list: "order-service is currently at these 3 addresses."

🏚️ The Monolith Reality
The monolith never had this problem: one process, one fixed address, done. The moment Order Service needs to call Payment Service as a separate app running on a separate host, a naive first cut hard-codes Payment's IP into Order's config. That works for exactly as long as Payment's IP never changes — which, once you're running on auto-scaling cloud infrastructure, is measured in hours, not months.
🚧 Why Hard-Coded IPs Can't Survive Decomposition
Splitting into services only pays off if those services can scale and redeploy independently of each other — but that same independence is exactly what breaks static addressing:
  • Auto-scaling changes IPs constantly: a sale event might add 27 new Payment pods in two minutes — 27 addresses that didn't exist a moment ago.
  • Deploys change IPs on every release: a rolling deploy replaces every pod with a fresh one at a fresh address, potentially several times a day.
  • Crashes must be invisible to callers: if a pod dies, every caller with its IP hard-coded keeps sending requests into a black hole until someone manually intervenes.
🔀 How Microservices Solve It
Instances register themselves with a registry on startup and send heartbeats. Callers ask the registry for "payment-service" by name and get back only the currently healthy addresses — scale-ups, deploys, and crashes are absorbed by the registry, invisible to every caller.
Instances register themselves; callers ask the registry, never hard-code IPs.
🌍 Proven at Scale
  • Tinder famously migrated its backend from a few hundred VMs running a large service footprint into hundreds of microservices on Kubernetes, as documented in Kubernetes' own public case study. The move only worked because Kubernetes gives every service a stable DNS name backed by live, auto-updated endpoints — matching pods appear and disappear behind that name constantly (deploys, autoscaling, node failures) without any service needing to track individual pod IPs.
  • Uber's dispatch system geo-shards services by city/region and scales each shard independently as demand shifts through the day — this only works because services resolve each other by name through a discovery layer; the actual instance count and placement changes continuously as Uber scales regions up and down.

05

Circuit Breaker

Resilience

When a downstream service keeps failing, stop calling it and fail fast. Otherwise every request waits for a timeout, threads pile up stuck waiting, and your service dies too — a single sick dependency cascades into a platform-wide outage.

🏚️ The Monolith Reality
A monolith mostly can't have this failure mode — a slow function call inside one process is still just a slow function call; it doesn't have a network hop, a timeout, or a separate thread pool to exhaust. The moment Order Service calls Payment Service over the network instead of as a local method call, a slow or hanging dependency turns into a very different kind of problem: threads blocked waiting on I/O, not slow logic.
🚧 Why Splitting Into Services Creates Cascading Failure
Every network call between services is a new opportunity for one dependency's problem to become everyone's problem:
  • Thread pools are finite: if the bank's API hangs for 30 seconds per call, every Payment Service thread eventually sits blocked waiting on it.
  • The blockage propagates upstream: Order Service calls the now-frozen Payment Service and its own threads block too — one dependency's outage climbs the call chain.
  • Nothing in a plain HTTP client stops this on its own: without an explicit breaker, retries and timeouts alone can even make it worse by piling more load onto an already-struggling dependency.
🔀 How Microservices Solve It
Wrap inter-service calls in a circuit breaker. After a threshold of failures, the circuit trips OPEN and new calls fail in milliseconds via a fallback — instead of hanging — freeing threads immediately. Periodic test calls (HALF-OPEN) check if the dependency has recovered before fully trusting it again.
CLOSED = healthy. OPEN = tripped, fail fast. HALF-OPEN = careful probing.
🌍 Proven at Scale
  • Netflix is where this pattern was proven at scale: after their move to AWS exposed just how easily one degraded backend service could cascade into wider outages, Netflix built and open-sourced Hystrix — the circuit breaker library that established this exact CLOSED → OPEN → HALF-OPEN model as the industry standard (Resilience4j is its modern successor).
  • Flipkart, during Big Billion Days-scale sale traffic, depends on external bank/UPI switches for payment authorization that can slow down under the load every bank sees during a national sale. A circuit breaker around those calls is what lets checkout degrade gracefully (queue the payment, offer Cash on Delivery) instead of every Payment Service thread hanging and taking browsing and cart down with it.
💻 Complete Code
BankClient.java — @CircuitBreaker with a fallback
// dependency: io.github.resilience4j:resilience4j-spring-boot3
@Service
public class BankClient {

    private final RestTemplate restTemplate;

    public BankClient(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    @CircuitBreaker(name = "bankApi", fallbackMethod = "chargeFallback")
    public PaymentResult charge(PaymentRequest request) {
        return restTemplate.postForObject(
                "https://bank.example.com/charge", request, PaymentResult.class);
    }

    // Called automatically when the circuit is OPEN or the call fails.
    private PaymentResult chargeFallback(PaymentRequest request, Throwable t) {
        return PaymentResult.pending("Bank is busy. Payment queued, will retry.");
    }
}

06

Saga Pattern

Data

One business action often touches many services, each with its own database — so a single ACID transaction is impossible. A saga runs the action as a sequence of local steps, each with a compensating action to undo it if a later step fails.

🏚️ The Monolith Reality
The monolith's superpower: placing an order, charging a card, and reserving stock could all live inside one @Transactional method against one database — if inventory reservation failed, the whole transaction rolled back automatically, no exceptions. Giving each service its own database — Order Service's own Postgres, Payment Service's own Postgres, Inventory Service's own Postgres — is what makes this impossible: those three steps now live in three separate databases that no single transaction can span.
🚧 Why "Just Use a Distributed Transaction" Doesn't Work
Two-phase commit (2PC) sounds like the obvious fix, but it fails for structural reasons that database-per-service creates:
  • It requires every participant to lock resources and wait for a coordinator — under real traffic, that's slow and blocks other transactions.
  • Most modern data stores don't support it: Kafka, Cassandra, DynamoDB, and most NoSQL stores have no 2PC protocol to join.
  • A coordinator crash mid-commit can leave participants locked indefinitely — trading one failure mode for a worse one.
🔀 How Microservices Solve It
Replace one distributed transaction with a chain of local transactions plus compensations. Order Service commits locally, Payment Service commits locally, Inventory Service commits locally — if the last step fails, previously completed steps are undone in reverse order by explicit compensating calls (refund, cancel), not a database rollback.
Step 3 fails → refund payment → cancel order. Every step ships with its undo.
🌍 Proven at Scale
  • Uber books a ride as exactly this kind of chain: assign a driver → authorize payment → start the trip. Each step is a separate service with its own database, and each has an explicit undo — if payment authorization fails after a driver is assigned, that assignment is released (compensated) rather than left dangling. There is no single database transaction spanning driver-matching and billing at Uber's scale; there can't be.
  • Flipkart-style e-commerce checkout during a sale runs the same order → payment → inventory chain — the specific failure mode of "last unit sold to someone else a second earlier" is a saga compensation in action: refund the charge, cancel the order, show "just sold out" instead of silently losing track of the customer's money.
💻 Complete Code
OrderSagaOrchestrator.java — orchestration style (one brain directs the steps)
@Service
public class OrderSagaOrchestrator {

    private final OrderService orderService;
    private final PaymentClient paymentClient;
    private final InventoryClient inventoryClient;

    public OrderSagaOrchestrator(OrderService orderService,
                                 PaymentClient paymentClient,
                                 InventoryClient inventoryClient) {
        this.orderService = orderService;
        this.paymentClient = paymentClient;
        this.inventoryClient = inventoryClient;
    }

    public OrderResult placeOrder(OrderRequest request) {
        Order order = orderService.createPending(request);        // local txn 1

        PaymentResult payment;
        try {
            payment = paymentClient.charge(order);                // local txn 2
        } catch (Exception e) {
            orderService.markCancelled(order.getId());            // undo 1
            return OrderResult.failed("Payment failed");
        }

        try {
            inventoryClient.reserve(order);                       // local txn 3
        } catch (Exception e) {
            paymentClient.refund(payment.getTxnId());              // undo 2
            orderService.markCancelled(order.getId());             // undo 1
            return OrderResult.failed("Out of stock — amount refunded");
        }

        orderService.markConfirmed(order.getId());
        return OrderResult.success(order.getId());
    }
}
// In production these steps usually run via Kafka commands/events (async),
// and saga state is persisted so it survives a crash mid-way.
🧰 Why This Tech Stack
FlavorHow it works
Orchestration easier to reason aboutOne central conductor tells each service what to do next and triggers undo steps. Tools: Temporal, Camunda, Axon, or a custom Spring service + Kafka.
ChoreographyNo conductor — each service reacts to the previous event. Simple for 2–3 steps, hard to follow for many.
⚠️ Common interview trap: "Why not just use a distributed transaction (2PC)?" See the bottleneck above — it's slow, blocks under failure, and most modern stores don't support it. Sagas trade instant consistency for availability: the system is briefly "in between," but always ends up correct.
🎯 Interview one-liner: "A saga replaces the one impossible cross-database transaction that database-per-service creates with a sequence of local transactions plus compensating actions — I prefer orchestration for complex flows because the whole state machine lives in one visible place."
07

Event-Driven Communication

Messaging

Instead of Service A directly calling B, C, and D, it announces "this happened" on a broker like Kafka. Whoever cares, listens — A never knows or cares who's subscribed.

🏚️ The Monolith Reality
The monolith's version: saving a post and updating the feed, notifications, and analytics were often just sequential method calls inside one request — slow, but simple, and everything happened in one transaction. Once these become separate services, each reachable only over the network, the naive translation is Post Service synchronously calling Feed, Notification, and Analytics services in turn, one HTTP call after another, before it can respond to the user.
🚧 Why Synchronous Calls Don't Scale Across Services
Chaining synchronous calls across services couples them in time and in knowledge — the opposite of what decomposition was for:
  • The slowest call in the chain holds everything hostage: if Notification Service is slow, the user's "post" spinner waits for it too, even though notifications aren't essential to posting.
  • Any one failing fails the whole action: Analytics Service being briefly down shouldn't mean photos can't be posted — but with a synchronous chain, it does.
  • Adding a new consumer means editing the producer: plugging in a new ML tagging service means Post Service's code and deploy have to change, re-coupling teams that decomposition was supposed to separate.
🔀 How Microservices Solve It
Publish one event and let interested services subscribe independently. Post Service saves the post, publishes PostCreated to Kafka, and responds to the user in milliseconds — Feed, Notification, Analytics, and any future consumer process it in parallel, on their own time, with zero code changes required upstream.
One event, many independent listeners. Adding a listener costs the publisher nothing.
🌍 Proven at Scale
  • Instagram-style apps: tapping Share triggers one PostCreated event. Feed Service pushes it into followers' timelines, Notification Service pings close friends, Media Service generates thumbnails, and Analytics counts it — five teams, zero synchronous coordination, and any one being briefly down never blocks the upload.
  • Google Maps' live traffic layer is a huge-scale version of the same idea in reverse: millions of phones stream anonymized location pings continuously. Maps doesn't process each ping with a synchronous call back to the sending phone — it ingests them as a firehose of events into a streaming pipeline that aggregates speed and congestion per road segment. A synchronous request/response model could never absorb that volume; publish-and-forget streaming is the only shape that fits.

@Service
public class OrderEventPublisher {

    private final KafkaTemplate<String, OrderPlacedEvent> kafkaTemplate;

    public OrderEventPublisher(KafkaTemplate<String, OrderPlacedEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void publishOrderPlaced(Order order) {

        // key = orderId → all events of one order land in the same partition, in order

    }
}

Transactional Outbox

Messaging

The dual-write problem: saving to your database AND publishing to Kafka are two separate systems, so one can succeed while the other fails. Fix: write the event into an outbox table in the same DB transaction as your data; a separate process reliably ships it to Kafka.

🏚️ The Monolith Reality
The naive event-publishing code — save to the database, then publish to Kafka — looks like: db.save(debit) then kafka.send(event) as two separate lines. A pod killed by an OOM between those two lines leaves the database updated but the event never sent — a wallet debited with no downstream service ever told about it. This exact gap didn't exist in the monolith because there was no second system to fall out of sync with; a single local transaction covered everything.
🚧 Why "Just Publish After Saving" Isn't Safe
The database and Kafka are two independent systems with no shared transaction — the dual-write problem is unavoidable without a specific fix:
  • A crash between the two writes loses the event silently — the database change is real, but nothing downstream ever hears about it.
  • Reordering the writes doesn't help: publish-then-save risks the opposite failure — an event about something that never actually got persisted.
  • At real transaction volumes, even a 0.01% loss rate means hundreds of silently broken records a day, each needing manual reconciliation.
🔀 How Microservices Solve It
Write the event into an outbox table, in the same local database transaction as the actual data change. A poller (or Debezium reading the DB's transaction log) picks up outbox rows afterward and publishes them to Kafka, marking them sent only after a successful publish — so the event can never be lost, only occasionally duplicated — and a consumer designed to safely ignore a duplicate message (the Idempotency pattern, later in this guide) makes that harmless.
Data + event committed atomically in one DB; a poller ships events to Kafka afterwards.
🌍 Proven at Scale
"At-least-once delivery plus idempotent consumers" — exactly what the outbox pattern gives you — is how real payment infrastructure (Stripe, PayPal, UPI switches) guarantees an event is never lost. A duplicate notification is a minor annoyance; a wallet debit that Ledger never heard about is a support escalation and, at scale, a compliance problem. This is why banking-grade systems treat the outbox as non-negotiable the moment money crosses a service boundary — not just a nice-to-have.
💻 Complete Code
OrderService.java + OutboxPublisher.java
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final OutboxRepository outboxRepository;
    private final ObjectMapper objectMapper;

    // ONE local ACID transaction — both rows, or none. No dual-write problem.
    @Transactional
    public Order placeOrder(OrderRequest request) throws JsonProcessingException {
        Order order = orderRepository.save(Order.confirmed(request));

        OutboxEvent event = new OutboxEvent(
                "Order", order.getId().toString(), "OrderPlaced",
                objectMapper.writeValueAsString(order));
        outboxRepository.save(event);                      // same DB, same txn

        return order;
    }
}

@Component
public class OutboxPublisher {

    private final OutboxRepository outboxRepository;
    private final KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedDelay = 2000)
    public void publishPendingEvents() {
        List<OutboxEvent> pending = outboxRepository.findTop100ByStatusOrderByCreatedAt("PENDING");

        for (OutboxEvent event : pending) {
            kafkaTemplate.send("order-events", event.getAggregateId(), event.getPayload())
                         .whenComplete((result, ex) -> {
                             if (ex == null) {
                                 event.setStatus("SENT");   // mark done only after Kafka ack
                                 outboxRepository.save(event);
                             }
                             // on failure: stays PENDING → retried next cycle
                         });
        }
    }
}

Centralized Logging & Distributed Tracing

Observability

One user click can now travel through 6 services. Tracing gives every request a trace ID that follows it everywhere; centralized logging puts all logs in one searchable place, tagged with that ID.

🏚️ The Monolith Reality
Debugging the monolith meant one log file, one stack trace, one grep. "Checkout is slow" meant reading top-to-bottom through one process's logs. Once checkout is decomposed across a gateway, an order service, a payment service, and a bank integration, that single stack trace is gone — the request's story is now scattered across 6 different machines' log files, each with its own local timestamp and no shared identifier connecting them.
🚧 Why Grepping Six Log Files Doesn't Scale
Decomposition trades one easy-to-read story for six disconnected fragments:
  • No shared identifier means no way to reassemble the story: which of the 40,000 "payment failed" lines in Payment Service's log corresponds to the specific slow checkout a customer reported?
  • SSH-and-grep doesn't scale past a handful of services: at 6 services it's painful; at Netflix or Uber's hundreds of services, it's simply impossible by hand.
  • Slowness is now a distributed question: "which of the 6 hops actually took the 4 seconds?" has no answer without visibility into every hop's timing, together.
🔀 How Microservices Solve It
Attach one trace ID to a request at the gateway and propagate it through every hop (HTTP headers, Kafka message headers). Every service tags its log lines with that ID and ships them to one central store; a tracing backend renders the whole journey as a waterfall, showing exactly which hop was slow.
Same trace ID at every hop → a waterfall view shows exactly where time went.
🌍 Proven at Scale
At Netflix or Uber scale — hundreds of services, millions of requests per minute — SSH-and-grep debugging isn't a slower option, it's not an option at all. Both companies (and most large microservice operators) rely on exactly this trace-ID-plus-central-logging model: an engineer filters traces by duration and service in a tracing UI, jumps straight to the slow hop, then pivots to that hop's logs using the same trace ID — turning what used to be hours of cross-team investigation into minutes, without ever SSH-ing into a single box.

Containerization & Orchestration

Infra

A container (Docker) packs a service + runtime + config into one standard box that runs the same everywhere. An orchestrator (Kubernetes) manages hundreds of these boxes: starts them, restarts crashed ones, and scales each one independently.

🏚️ The Monolith Reality
Deploying the monolith meant provisioning one kind of server and scaling it by adding identical copies behind a load balancer — simple, because there was only one thing to run. Once decomposed into 15+ independently-deployable services, each with different runtime needs, dependency versions, and scaling profiles, manually provisioning and babysitting VMs for each one becomes an operational nightmare that grows with every new service.
🚧 Why Manual VM Management Can't Keep Up
Independent deployability — the entire reason services were split apart in the first place — is worthless if deploying each one is manual, slow, and inconsistent:
  • "Works on my machine" multiplies by N services: each service can drift onto a slightly different JDK patch or OS library version across environments.
  • Manual scaling can't react in minutes: a flash sale needs Order Service to go from 3 to 30 instances in two minutes — no human can provision VMs that fast, repeatedly, every day.
  • Nobody's watching 15+ services' health at 2 AM: a crashed instance needs to be detected and replaced automatically, or someone's phone rings every night.
🔀 How Microservices Solve It
Package every service identically as a container image, and let Kubernetes manage the fleet. The image runs the same in dev, staging, and production. Kubernetes' autoscaler watches metrics and adjusts replica counts per service; liveness/readiness probes detect and replace crashed instances automatically — no human paged for a routine pod crash.
K8s heals crashed pods and scales the hot service automatically.
🌍 Proven at Scale
  • Flipkart's Big Billion Days sale is a public, well-documented case of exactly this: traffic and order volume jump to many multiples of a normal day within minutes of the sale opening. Manually provisioning enough VMs in advance for peak while not paying for that capacity the other 360 days a year isn't practical — Kubernetes-style autoscaling lets Order and Search services scale up right as the sale opens and back down after, while catalog-browsing services barely move.
  • Netflix runs thousands of services on a container-orchestrated fleet for exactly the same reason: streaming demand varies enormously by time zone and by evening peak, and no team wants to hand-manage instance counts for hundreds of services around the clock.

SQL vs NoSQL — Picking the Right Database

Data

There's no "best" database — each type is shaped for a different data shape and access pattern. Since giving each service its own database lets every service choose independently, a healthy microservices system usually runs several database types at once — polyglot persistence.

🏚️ The Monolith Reality
The monolith usually ran one database for everything because that's what "one deployable unit" implied — orders, product catalog, sessions, and activity logs all forced into the same relational schema. The catalog team wanted flexible per-category attributes; the ops team wanted to store session data that didn't need durability at all; both got the same rigid relational tables because there was only one database to argue about.

🚧 Why One Database Can't Fit Every Access Pattern
Forcing every access pattern onto one engine means every workload except the one the schema was designed for suffers:
  • Flexible-schema data becomes dozens of nullable columns: a TV needs screen size, a shirt needs fabric — one relational table for "products" ends up mostly empty per row, or needs a schema migration every time a new category launches.
  • Grab-by-key workloads pay for query planning they don't need: a session lookup is always "give me THE cart for user X" — running that through a full SQL query planner is wasted overhead at millions of requests per minute.
  • Firehose write volume doesn't fit a single-writer relational instance: billions of "user viewed X" events a day will outpace what one primary database node can absorb, no matter how well-indexed it is.
🔀 How Microservices Solve It
Let each service pick the storage engine that matches its actual queries — strict relational for money and rules, key-value for grab-by-key, document for flexible schemas, wide-column for massive partitioned writes, graph for relationship traversal. Database-per-service is what makes mixing all five safely possible in one system.
Five questions, five shapes, five different databases — all in one system.
🌍 Proven at Scale
  • Amazon's DynamoDB was born directly from this bottleneck: their shopping cart service hit relational database scaling limits during peak Christmas traffic in the early 2000s, which pushed Amazon to build a key-value store purpose-built for "grab-by-key, millions of times a minute" — a workload a relational engine handles poorly at that volume.
  • Netflix writes viewing-history events into Cassandra specifically because the write volume (billions of events/day, always reads-by-user-ID) needs a store that spreads writes across hundreds of machines and tolerates node failures without blocking writes.
  • LinkedIn-style "People you may know" style features are the classic case for a graph database (Neo4j): "friends of friends" is a natural relationship walk in a graph, but a self-join nightmare in SQL.

Caching with Redis

Performance

Keep frequently-read data in memory, right next to the service, so most reads never touch the database. RAM answers in under a millisecond and, just as important, shields the database from millions of identical queries.

🏚️ The Monolith Reality
A monolith's single database absorbed everything because traffic was usually modest enough that direct queries were fine. But once one service (say, Profile Service) is handling millions of reads for the same rows — a celebrity's profile, a trending product — the "just query the database every time" approach that worked at monolith scale becomes the exact bottleneck decomposition was supposed to relieve.
🚧 Why "Just Query the Database" Stops Working
Splitting services apart doesn't reduce read volume on hot data — if anything, it concentrates it onto one smaller service:
  • Popularity creates self-inflicted load spikes: 5 million people opening the same profile in 10 minutes means 5 million nearly-identical queries hitting the same rows.
  • The database's connection pool is finite: enough concurrent identical queries exhausts connections, and every other feature sharing that database slows down too.
  • Read replicas only buy you a multiple, not headroom: at true viral scale, even replicated reads eventually saturate — the problem is query volume, not just where the data lives.
🔀 How Microservices Solve It
Cache-aside with Redis: check the cache first; on a miss, read the database once and populate the cache with a TTL. The database ends up seeing roughly one query per key per TTL window instead of millions — regardless of how popular that key becomes.
Check cache → on miss, read DB once → store with TTL. The DB sees ~1 query per key per minute.
🌍 Proven at Scale
  • Instagram-scale apps: when a celebrity posts, millions of identical profile-page requests hit within minutes. Without a cache, that's millions of identical SQL queries — enough to exhaust the database's connections and slow down every unrelated feature on the platform. Cache-aside turns that into one DB read per 60-second TTL window.
  • Tinder's swipe deck — the batch of nearby candidate profiles served to a user — is a natural fit for the same pattern: recomputing "who's nearby and swipeable" from scratch on every single swipe would hammer the database at Tinder's request volume, so a precomputed deck is cached and served from memory, refreshed periodically rather than on every swipe.

CQRS — Separate Read & Write Models

Architecture

Writes (Commands) go into a strict, normalized model. Reads (Queries) are served from a separate, pre-flattened model kept up to date by events — useful when reads and writes have very different shapes or scale.

🏚️ The Monolith Reality
In the monolith, "My Orders" was one SQL join across order, product, and shipping tables in the same database — slow at real scale, but at least possible in one query. Giving Order, Product, and Shipping each their own database makes that join illegal: order, product, and shipping data now live in three separate databases owned by three separate services, and a single SQL join across them simply doesn't exist anymore.
🚧 Why Runtime Joins Across Services Don't Scale
The natural first fix — have the "My Orders" page call all three services live, every time — reintroduces exactly the coupling and fragility decomposition was meant to remove:
  • Three network calls per page view, at millions of views per hour, multiplies load and latency compared to one query.
  • Any one dependency being down breaks the whole page: Product Service having a bad day shouldn't mean your order history is unreadable.
  • Stitching results client-side or in a thin aggregator duplicates logic every page that needs this view has to repeat.
🔀 How Microservices Solve It
Build a separate, purpose-shaped read model. A small projector listens to events from Order, Product, and Shipping services and maintains one flattened "order history" document per order in a fast read store. The page becomes a single, cheap query — no runtime joins, no dependency on three services being up simultaneously.
Writes stay strict on the left; events build a ready-to-show view; reads hit only the view.
🌍 Proven at Scale
Amazon's "Your Orders" page is the textbook version of this pattern: order details, product name/image, and live delivery status come from separate systems, but the page renders as one fast, flattened view rather than live cross-service joins. Instagram's feed — built the same way, via events published whenever someone posts — is the same idea at more extreme scale — write the post once, precompute the read side for every follower — because computing a personalized feed live, per page-load, at that traffic volume isn't feasible.

Idempotency — Safe Retries

Resilience

An operation is idempotent if doing it twice has the same effect as doing it once. Timeouts, retries, and at-least-once delivery — the guarantee a Transactional Outbox gives you — mean duplicates will happen — idempotency is what makes them harmless.

🏚️ The Monolith Reality
A single in-process call either fully happened or fully didn't — there was no network in between to drop the response. The moment Payment Service is a separate service reached over HTTP instead of a local method call, a client can get a timeout on a request that did succeed server-side — the bank actually charged the card, but the response never made it back. Retrying blindly, which feels like the obvious fix, now risks charging twice.
🚧 Why "Just Retry on Timeout" Is Dangerous
Once a call crosses the network, "did it work?" stops being a question the client can answer with certainty:
  • A timeout doesn't mean failure — the server may have completed the operation and only the response was lost.
  • Don't retry, and money may never move — the customer thinks they paid; nothing was actually charged.
  • Retry blindly, and money may move twice — exactly the failure mode a naive retry-on-timeout policy introduces.
🔀 How Microservices Solve It
Every state-changing endpoint takes a client-generated idempotency key. The server atomically claims the key (e.g. Redis SETNX) before doing the real work; a duplicate request with the same key gets the stored result replayed instead of a second charge.
Same key twice → second attempt gets the saved answer, not a second charge.
🌍 Proven at Scale
This is literally how Stripe's public payment API works — every POST accepts an Idempotency-Key header, for exactly the reason above: "should I retry a payment timeout?" has no safe answer without one. On the Kafka side, where a Transactional Outbox guarantees at-least-once delivery, consumers record processed event IDs and silently skip repeats — the same principle, applied to event processing instead of HTTP retries.
💻 Complete Code
PaymentService.java — claim the key, charge once, replay duplicates
@Service
public class PaymentService {

    private final StringRedisTemplate redis;

    public PaymentService(StringRedisTemplate redis) { this.redis = redis; }

    public PaymentResponse pay(String idempotencyKey, PaymentRequest request) {

        Boolean firstTime = redis.opsForValue()
                .setIfAbsent("idem:" + idempotencyKey, "IN_PROGRESS", Duration.ofHours(24));

        if (Boolean.FALSE.equals(firstTime)) {
            String saved = redis.opsForValue().get("result:" + idempotencyKey);
            return PaymentResponse.fromJson(saved);   // duplicate → replay, no re-charge
        }

        PaymentResponse response = chargeBank(request);   // happens exactly once
        redis.opsForValue().set("result:" + idempotencyKey,
                                response.toJson(), Duration.ofHours(24));
        return response;
    }
}

Strangler Fig — Monolith to Microservices, Safely

Architecture

You almost never rewrite a monolith in one go. Put a gateway in front, carve out one capability at a time into a new service, route just that traffic to it, and repeat — until the monolith is an empty shell you switch off.

🏚️ The Monolith Reality
Every concept in this guide assumes services already exist — but real companies start with exactly the ten-year-old monolith this guide has been contrasting against, and it's still running the whole business while you read this. Rewriting it from scratch is tempting and almost always a trap: an 18-month rewrite means 18 months of zero user-facing value shipped, while the old system keeps changing underneath you the whole time — you're chasing a moving target with no interim payoff.
🚧 Why a Big-Bang Rewrite Fails in Practice
A full rewrite concentrates all the risk this entire guide has been about spreading out:
  • One giant cutover day instead of many small, reversible ones — if it fails, everything fails at once.
  • The business doesn't pause for you: the monolith keeps getting feature changes and bug fixes during the rewrite, so the target keeps moving.
  • No incremental validation: you don't find out the new architecture has a problem until the very end, at maximum cost to fix it.
🔀 How Microservices Solve It
Strangler fig migration: put an API gateway in front of the monolith first (changes nothing else). Then carve out one capability, route just its traffic to the new service, and validate before moving to the next. Rollback is always one gateway route flip away. The monolith shrinks like the host tree inside a strangler fig, until one day it's switched off entirely.
The gateway peels traffic away route by route; the monolith shrinks until it's gone.
🌍 Proven at Scale
  • Amazon famously did not rewrite its early monolith overnight — the internal API mandate — forcing every team to expose functionality only through APIs, the same root idea behind splitting into services in the first place — forced capabilities apart gradually, capability by capability, over years, while the business kept shipping.
  • Netflix spent roughly the better part of a decade migrating from its data-center monolith to cloud-based microservices, carving out services piece by piece and running old and new side by side during the transition rather than attempting a single cutover.
  • The graveyard of "we'll rewrite everything from scratch in 18 months" projects across the industry is far larger than the list of companies that pulled it off — which is exactly why the strangler fig approach is the default recommendation, not the rewrite.
